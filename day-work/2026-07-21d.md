## 10. Mobile login endpoint + Bearer JWT on seller routes

### Goal

Additive-only path for React Native / Expo: plain JWT stored on device and sent as `Authorization: Bearer <token>`. **Do not** change web cookie Credentials flow, `src/auth.ts`, or `src/auth.config.ts`. No school-email restriction or student ID verification.

### Dependencies installed

```text
jsonwebtoken            → (runtime) sign mobile tokens
@types/jsonwebtoken     → (dev)
jose                    → direct dep for Edge-safe JWT verify in middleware
```

### Files created

| File | Summary |
|---|---|
| `src/app/api/auth/mobile-login/route.ts` | `POST` body `{ email, password }` → lookup User by email; if no user / `password` null / bcrypt fail → **401** `{ error: "Invalid email or password" }` (same message always, no email enumeration); on success sign JWT and return **200** `{ token, user: { id, email, name, role } }` — never returns password hash |
| `src/lib/sign-mobile-token.ts` | Node-only: `jwt.sign({ userId, role }, AUTH_SECRET, { expiresIn: "30d" })` via **jsonwebtoken** |
| `src/lib/mobile-auth.ts` | Edge-safe helpers: `extractBearerToken(authorization)`, `verifyMobileToken(token)` via **jose** `jwtVerify` + `AUTH_SECRET` (HS256, same secret as sign); returns `{ userId, role }` or `null` |

### Files modified

| File | Change |
|---|---|
| `src/middleware.ts` | Either/or auth on seller-only routes (`POST /api/listings`, `/seller/*`): (1) if `Authorization: Bearer …` present → verify mobile JWT → **401** invalid/expired, **403** valid but `role !== SELLER`, else allow; (2) else existing NextAuth cookie session path unchanged (**401** / **403** same as before). Web and mobile do not replace each other. |
| `package.json` / `package-lock.json` | Added `jsonwebtoken`, `@types/jsonwebtoken`, `jose` |

### Design notes

- **Sign:** `jsonwebtoken` in Node route handler only (`sign-mobile-token.ts`).
- **Verify in middleware:** Next.js middleware is **Edge**, so `jsonwebtoken` cannot be imported there. Split modules: middleware imports only `mobile-auth.ts` (**jose**). Tokens remain standard HS256 JWTs signed with `AUTH_SECRET` — compatible.
- **Payload:** `{ userId, role }` (+ `iat` / `exp` from jsonwebtoken). Expiry **30 days**.
- **Untouched:** `src/auth.ts`, `src/auth.config.ts`, web Credentials cookie login, `/login` page (not built), school-email restriction, student ID verification.

## 12. Typed error UI (status-colored) + waitlist retry messaging

### Goal

Recognize error kinds (especially **5xx / Neon cold start** and network failures), show a **designed** error surface with **bg/border colors by type**, and push users to **retry** when appropriate. Reusable beyond waitlist.

### Files created

| File | Summary |
|---|---|
| `src/lib/api-error.ts` | `AppError` + `ErrorVariant`; `appErrorFromStatus`, `appErrorFromResponse`, `appErrorFromNetwork`. Maps 400/401/403/404/409/5xx/network → `{ variant, status, title, message, retryable, codeLabel }`. | `src/components/ui/error-message.tsx` | `ErrorMessage` component (CVA variants): icon, title, mono code badge, body, optional “Tap the button below to try again.” when `retryable`, optional dismiss. |

### Color / type map (UI)

| Variant | Typical HTTP | Background / border |
|---|---|---|
| `validation` | 400 | Amber `bg-amber-50` / `border-amber-300` |
| `unauthorized` | 401 | Orange `bg-orange-50` / `border-orange-300` |
| `forbidden` | 403 | Rose `bg-rose-50` / `border-rose-300` |
| `notFound` | 404 | Sky `bg-sky-50` / `border-sky-300` |

| `network` | (no status) | Slate `bg-slate-100` / `border-slate-300` |

| File | Change |
